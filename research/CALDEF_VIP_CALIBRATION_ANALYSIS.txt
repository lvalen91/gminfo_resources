1. EXECUTIVE SUMMARY
================================================================================

After comprehensive analysis of the GIS-646 CalDef Database and VIP log files,
a critical architectural insight has been discovered:

  THE GM AAOS SYSTEM HAS TWO COMPLETELY SEPARATE CALIBRATION SYSTEMS:

  1. CalDef Database Calibrations -> Android/SoC software (Palto)
  2. VIP Firmware Calibrations -> VIP MCU (RH850)

The EEPROM security flags (0x0440, 0x0A80 for ADB) are controlled by VIP
firmware, NOT by CalDef calibrations. Therefore, modifying CalDef calibrations
via DPS calibration files (.cfx) CANNOT affect EEPROM security flags.

2.2 IMPLICATION
---------------
CalDef calibrations like DEVICE_REGISTRATION_ENABLE_CHECK_OVERRIDE affect
Android software behavior (MEC check bypass for registration), but DO NOT
affect VIP's EEPROM security flag validation.

The PROTOKEY/ADB authentication is handled by VIP firmware reading EEPROM
flags directly via I2C - this is completely independent of CalDef.

3.0 CALDEF RECORD STRUCTURE (migrated from DPS_SECURITY_ANALYSIS.txt)
---------------------------------------------------------------------
Each calibration record has:
  - CalID (unique identifier)
  - Type (Boolean, Integer, String, Enumeration)
  - Default value
  - Reference (GIS specification)
  - Usage (LOW, MID, HIGH, GLOBALA, GLOBALB, VCU, etc.)
The "Usage" field indicates which vehicle/radio tiers use the calibration, NOT which
Module ID contains it.

CalDef limitations: CalDef files define WHAT calibrations exist; they do NOT tell us
which Module ID contains each calibration, the binary format of calibration storage,
how to create valid .cfx files, or whether calibrations are signed.

3.1 SECURITY-RELEVANT CALDEFS
-----------------------------

A) DEVICE_REGISTRATION_ENABLE_CHECK_OVERRIDE (SFS2)
   File:    SFS2_SecurityExtensionsFRD.caldef
   Type:    Boolean
   Default: FALSE
   Effect:  Bypasses MEC check for device registration
   Scope:   Android Palto software (NOT VIP)

   IF TRUE:  Device registration proceeds regardless of MEC state
   IF FALSE: Registration waits for MEC == 0

   IMPORTANT: This does NOT affect ADB authentication!
              ADB auth is VIP firmware (PROTOKEY + EEPROM flags)

B) EXPIRATION_ENFORCEMENT (SFS2)
   File:    SFS2_SecurityExtensionsFRD.caldef
   Type:    Boolean
   Default: TRUE
   Effect:  Controls certificate expiration validation
   Scope:   Android TCS library

   IF FALSE: May accept expired certificates
   IF TRUE:  Enforces certificate expiration (normal mode only)

C) GIS763_PROGRAMMING_ENABLED (GIS763)
   File:    GIS763_InfotainmentProg_GlobalB.caldef
   Type:    Boolean
   Default: TRUE
   Effect:  Master programming enable switch
   Scope:   Android update subsystem

D) GIS763_SigningCertificateIssuer (GIS763)
   Type:    String
   Default: "CN=GPD, OU=GMNA GPD PRODUCTION, O=GENERAL MOTORS LLC"
   Effect:  Expected certificate issuer for signature validation

E) GIS763_SigningCertificateSubject (GIS763)
   Type:    String
   Default: "CN=GM Code Signature Server, OU=GM NA INFOTAINMENT, O=GM"
   Effect:  Expected certificate subject for code signing

3.2 OTHER CALDEF FILES (100+ files)
-----------------------------------
Categories:
- UI/HMI calibrations (GIS500-599)

4.2 VIP CALIBRATION STORAGE
---------------------------
These calibrations are stored in VIP internal flash/NVM.
They can ONLY be modified by:
1. Flashing new VIP_APP firmware via DPS (signed)
2. Potentially via VIP JTAG debugging (physical access)

DPS cannot modify these calibrations without replacing VIP firmware.

5.3 CRITICAL OBSERVATION
------------------------
The SoC requests calibration data from VIP via IPC.
VIP reads this data from its internal storage and sends to SoC.

This is a ONE-WAY flow: VIP -> SoC
The SoC cannot write calibration data back to VIP via IPC.

6. SECURITY-RELEVANT CALIBRATIONS
================================================================================

6.1 WHAT CONTROLS ADB ACCESS?
-----------------------------
ADB access is controlled by:

1. EEPROM Flag 0x0440: Primary ADB enable (5A FF 5A = enabled)
2. EEPROM Flag 0x0A80: Backup ADB enable (5A FF 5A = enabled)
3. VIP PROTOKEY module: Validates BCM seeds for ADB auth

When EEPROM flags are set to bypass:
- VIP returns 0xFF seed (all bytes) instead of real challenge
- PROTOKEY validation is skipped
- ADB access is granted without authentication

6.2 WHAT CALDEF CALIBRATIONS AFFECT?
------------------------------------
| Calibration | Affects | Does NOT Affect |
|-------------|---------|-----------------|
| DEVICE_REGISTRATION_ENABLE_CHECK_OVERRIDE | Device registration | ADB auth |
| EXPIRATION_ENFORCEMENT | Certificate validation | ADB auth |
| GIS763_PROGRAMMING_ENABLED | OTA/USB programming | ADB auth |
| SECURITY_EVENT_LOGGING_ENABLED | Security logging | ADB auth |

NONE of the CalDef calibrations affect ADB authentication.

7. EXPLOITATION ANALYSIS
================================================================================

7.1 CAN DPS CALIBRATION FILES ENABLE ADB?
-----------------------------------------
ANSWER: NO

DPS calibration files (.cfx) modify Android/SoC calibrations.
These calibrations do not control VIP EEPROM access.
Therefore, creating custom .cfx files cannot enable ADB.

7.2 CAN DPS MODIFY VIP CALIBRATIONS?
------------------------------------
ANSWER: ONLY VIA VIP FIRMWARE FLASH

VIP calibrations are embedded in VIP_APP firmware.
To modify them, you would need to:
1. Extract VIP firmware
2. Modify calibration values
3. Re-sign firmware (requires GM signing keys)
4. Flash via DPS

Without GM signing keys, VIP firmware modification is blocked.

7.3 REMAINING ATTACK VECTORS
----------------------------
1. PHYSICAL EEPROM MODIFICATION (Proven working)
   - Direct I2C access to M24C64
   - Modify 0x0440 and 0x0A80

2. Y177 FIRMWARE DOWNGRADE
   - SELinux permissive in Y177
   - More exploration possible
   - Requires testing AVB rollback

3. VIP UTILITY FILE ANALYSIS
   - If utility files contain I2C opcodes
   - Could potentially write EEPROM via DPS
   - Requires reverse engineering .dll format

4. VIP UART CONSOLE
   - /dev/ttyACM1 accessible
   - May accept debug commands
   - May require bypassing 74LVC08A gate

5. VIP JTAG
   - Physical debug access
   - Full firmware dump/modify
   - Requires hardware probing

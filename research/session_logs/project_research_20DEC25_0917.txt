ARCHIVAL SESSION LOG (Dec 20 2025) -- retained as history; body UNCHANGED. Several
hypotheses here were later RESOLVED/REFUTED. Do NOT trust the following as current:
  - "EEPROM 0x0B40/0x0B41 controls SELinux/debug mode" (L256, 341-346, 544, 571-576, 633):
    WRONG. SELinux mode is set by the kernel cmdline (Y181 Enforcing / Y177 permissive),
    NOT by EEPROM. 0x0B41->0x01 is part of the VERIFIED EEPROM ADB-UNLOCK set (alongside
    0x0441/0x0A81/0x1A01->0xFF); its independent function (debug / verbose logging) is a
    hypothesis, not SELinux control.
  - "Y177 downgrade may be possible" (L347-356, 438-451, 638): WRONG. Y181->Y177 is BLOCKED
    by the GHS hypervisor rollback counter (misc / vda9); tested FAILED.
  - "RAM: 6GB DDR3L" (L55-56): WRONG. 8GB LPDDR4 (4x Micron MT53E512M32D2DS-053), ~5.66GB
    kernel-visible.
For current verified facts see VERIFICATION.md and research/MASTER_REFERENCE.md.

3. SOFTWARE ARCHITECTURE
================================================================================

Boot Chain:
1. Power On
2. Intel ABL (Apollo Lake Bootloader)
3. GHS INTEGRITY Hypervisor starts
4. Hypervisor spawns safety-critical partition (RTOS)
5. Hypervisor spawns Android partition (guest OS)
6. Linux Kernel boots (under hypervisor)
7. Android AAOS userspace

Partition Layout (from /dev/block/by-name/):
- ghs_isys_a/b    : Green Hills INTEGRITY system
- ghs_abl_update  : ABL update staging
- ghs_storage     : GHS persistent storage
- ghs_log         : Hypervisor logs
- boot_a/b        : Android boot images
- bootloader_a/b  : Bootloader images
- system          : Android system (dynamic partition)
- vendor          : Vendor partition (dynamic partition)
- product         : Product partition (dynamic partition)
- vbmeta_a/b      : Verified boot metadata
- data            : User data (/dev/block/vda26)
- super           : Dynamic partition container
- misc            : Boot control
- metadata        : Encryption metadata
- persistent      : FRP data
- teedata         : TEE storage
- calibration     : Hardware calibration
- config          : Configuration

OTA Update System:
- Format: Full partition images (not differential)
- Method: Direct block writes (like dd)
- Script: Edify-like syntax with gmext.* extensions
- Commands:
  - gmext.write_emmc_image() : Direct partition write
  - gmext.extract_emmc_image() : Extract to partition
  - gmext.extract_logical_partition() : Dynamic partitions
  - gmext.update_ecu() : ECU firmware (VIP, GPS, SXM, etc.)
  - gmext.switch_slot() : A/B slot switching
- Both Y177 and Y181 have "version check disabled" in manifests

VIP App Module Files:

- Same size but 549,518 byte differences

Key Strings Found in VIP Firmware:

ICUSB (ADB) Control:
[PROTOKEY] ICUSB module disabled
[PROTOKEY] ICUSB module enabled
[AME_DIAG] ICUSB enabled and both key are provisioned.
[AME_DIAG] ICUSB enabled and no key provisioned.
[AME_DIAG] ICUSB enabled and Master key provisioned.
[AME_DIAG] ICUSB enabled and Unlock key provisioned.
[AME_DIAG] ICUSB not enabled.

EEPROM Default Restoration:
[SS_SWC] cal_xxx (%d) read eeprom error !, Reinstating the default value (%d)
[SS_SWC] cal_xxx (%d) read out of range !, Reinstating the default value (%d)

SoC Communication:
[J6_CDD] J6_prv_ProtoKey:Protokey transmitted to SoC, status is %d

Architecture Understanding:
1. VIP (Renesas) reads EEPROM for ICUSB flag
2. VIP validates value (range check, checksum)
3. If invalid → reset to default (disabled)
4. VIP sends ICUSB status to Intel SoC via IPC
5. SoC enables/disables ADB daemon based on VIP signal

   - /dev/ttyACM1 is world read/write from ADB shell

   - Test from ADB: cat /dev/ttyACM1

12. ADB BEHAVIOR FINDINGS
================================================================================

Observed Behavior:
1. EEPROM ADB mod enables capability (no RSA auth needed)
2. ADB works immediately after enabling in Developer Options
3. ON EVERY REBOOT: ADB setting is disabled
4. User must manually re-enable in Developer Options each boot
5. ADB sessions may timeout/disconnect during use

This Indicates:
- EEPROM controls CAPABILITY (can ADB be enabled)
- Software controls SETTING (is ADB currently enabled)
- Something resets the setting on every boot
- Likely an init script or GM service

Suspected Culprits:
- gmprotokey service (handles PROTOKEY/ICUSB)
- GMLoggerService
- Init.rc property trigger on boot_completed
- Security policy enforcement service

Workarounds:
- Enable ADB over WiFi as backup: adb tcpip 5555
- Use keepalive script to maintain connection
- Work quickly after enabling ADB
- Disable macOS USB power management

Investigation Script: find_adb_disable.sh

1. EXECUTIVE SUMMARY
================================================================================

Analysis of GM Infotainment 3 (IOK) module comparing Y177 and Y181 firmware
reveals significant security architecture changes and multiple potential attack
vectors. The critical finding is that SELinux enforcement was hardened between
versions, but several exploitable paths remain.

Key Findings:
- SELinux changed from PERMISSIVE (Y177) to ENFORCING (Y181)
- Boot parameter shows permissive but runtime is enforcing (override detected)
- ADB access achieved via EEPROM modification (0x0440 + 0x0A80)
- UART console (/dev/ttyACM1) is world-accessible
- Network ports 7000, 6363, 49156 are unexplored attack surface
- Y177 downgrade possible (version check disabled in manifest)
- EEPROM flag 0x0B40 suspected to control debug/SELinux mode

2. CRITICAL SECURITY FINDINGS
================================================================================

2.1 SELINUX CONFIGURATION ANOMALY
---------------------------------
Property Analysis:
  [ro.boot.selinux]: [permissive]   <- Kernel boot parameter
  getenforce output:  Enforcing     <- Runtime state

The kernel is booted with androidboot.selinux=permissive but something in the
boot chain FORCES enforcing mode. Likely causes:
  - init.rc trigger on boot completion
  - GM service (gmprotokey or gm_update_engine)
  - GHS INTEGRITY hypervisor enforcement

IMPLICATION: If the enforcing switch is controlled by software/EEPROM, it may
be bypassable.

2.2 ADB SECURITY BYPASS (CONFIRMED WORKING)
-------------------------------------------
| EEPROM Address | Stock Value | Modified Value | Effect              |
|----------------|-------------|----------------|---------------------|
| 0x0440         | C3 00 C3    | 5A FF 5A       | Primary ADB enable  |
| 0x0A80         | FF FF FF    | 5A FF 5A       | Backup ADB enable   |

Both addresses use 0x5A framing (security frame). Setting the flag byte to
0xFF disables GM Secure Client authentication requirement.

Frame Protocol:
  0x5A = Security/VIN frames
  0x69 = Configuration frames
  0xF0 = Data/String frames
  0xC3 = Calibration frames

2.3 POTENTIAL DEBUG/SELINUX EEPROM FLAG
---------------------------------------
| Address  | Frame | Current | Suspected Purpose           |
|----------|-------|---------|----------------------------|
| 0x0B40   | 0x69  | 0x00    | Developer/Debug mode toggle |
| 0x0B60   | 0xC3  | varies  | Unknown config              |

HYPOTHESIS: Setting 0x0B41 to 0xFF may enable debug mode or disable SELinux
enforcement. The 0x69 frame indicates a configuration toggle (same pattern
as other working flags).

10. CONCLUSION
================================================================================

The Y181 update significantly hardened security by enabling SELinux enforcement.
However, multiple attack vectors remain viable:

1. UART console via world-accessible /dev/ttyACM1
2. Y177 downgrade due to disabled version checking
3. EEPROM flags at 0x0B40 may control debug/SELinux mode
4. Network services on open ports are unexplored

MOST PROMISING PATHS:
---------------------
1. UART Investigation - Operates below Android layer, may provide hypervisor
   or bootloader access independent of SELinux state

2. EEPROM 0x0B40 Flag - Follows same pattern as working ADB flags, could
   restore permissive mode on Y181 without downgrade

3. Y177 Downgrade - Known working path to SELinux permissive, version check
   disabled, main risk is AVB rollback protection

CRITICAL INSIGHT:
-----------------
The boot property ro.boot.selinux=permissive combined with runtime Enforcing
state indicates the SELinux mode switch is SOFTWARE CONTROLLED, not hardware
enforced. This means there is likely a configuration option (EEPROM flag,
init trigger, or service) that controls this behavior and can be modified.

15.1 SELINUX ENFORCEMENT MECHANISM - ROOT CAUSE
------------------------------------------------
CRITICAL DISCOVERY: Boot.img cmdline contains DUPLICATE settings:

Early in cmdline:   enforcing=0 androidboot.selinux=permissive
Later in cmdline:   enforcing=1 androidboot.selinux=enforcing buildvariant=user

Linux kernel behavior: When a parameter appears multiple times, the LAST value wins.
Result: Both Y177 and Y181 boot into ENFORCING mode by design.

Boot Image Comparison:
| Component | Y177 | Y181 | Finding |
|-----------|------|------|---------|
| cmdline | enforcing=1 last | enforcing=1 last | IDENTICAL |
| init binary | MD5: 7786726c... | MD5: 7786726c... | IDENTICAL |
| sepolicy | MD5: e9ebb7d8... | MD5: 6b2045bf... | DIFFERENT |
| ramdisk size | 46,373,888 | 46,373,888 | Same |

IMPLICATION: The SELinux mode is NOT controlled by EEPROM or GM software.
It is hardcoded in the kernel cmdline. To achieve permissive:
  Option 1: Modify boot.img cmdline (requires signed image or AVB bypass)
  Option 2: Find kernel vulnerability to change runtime mode
  Option 3: Policy bypass through sepolicy differences

15.2 ADB DISABLE ON REBOOT MECHANISM
------------------------------------
Root Cause Analysis:

| Layer | Control | Setting | Effect |
|-------|---------|---------|--------|
| Ramdisk | persist.sys.usb.config | none | USB disabled at boot |
| Build Props | ro.debuggable | 0 | No auto-ADB enable |
| init.rc | on property:ro.debuggable=1 | Never triggers | Would enable ADB |
| VIP/EEPROM | 0x0440/0x0A80 | ICUSB flag | Hardware USB enable |

Boot Sequence:
1. Kernel boots with persist.sys.usb.config=none (from ramdisk)
2. This property is READ from ramdisk, overwriting any saved value
3. User build (ro.debuggable=0) skips auto-ADB trigger
4. ADB remains disabled until manually enabled via Developer Options
5. On reboot, step 1 repeats

WORKAROUND OPTIONS:
a. Modify ramdisk to set persist.sys.usb.config=adb (requires boot.img mod)
b. Enable ADB via Developer Options, hope it persists (doesn't work)
c. Use EEPROM ICUSB flag to enable hardware-level USB access

17.5 SECURITY BYPASS MECHANISM
------------------------------
Normal flow (GM Secure Client required):
  1. VIP reads ICUSB flag from EEPROM
  2. VIP receives seeds from BCM via GMLAN
  3. VIP transmits seeds to SoC via IPC (status=1)
  4. SoC/gm_protokey daemon requires valid GM Secure Client authentication
  5. ADB connection must authenticate with GM's RSA keys

Bypassed flow (EEPROM modified):
  1. VIP reads modified ICUSB flag from EEPROM
  2. VIP may or may not receive seeds (doesn't matter)
  3. VIP transmits to SoC, but bypass flag is set
  4. SoC/gm_protokey daemon allows unsecured connections
  5. ADB connection works without authentication

18.2 VULNERABILITY AT EACH LAYER
--------------------------------
| Layer | Component | Vulnerability |
|-------|-----------|---------------|
| EEPROM | M24C64 | No encryption, no authentication |
| VIP | PROTOKEY | Trusts EEPROM blindly, no tamper detection |
| HostOS | IPC | Forwards VIP data without verification |
| Android | gm_protokey | Trusts IPC data, grants access based on flag |

18.3 WHY THE BYPASS WORKS
-------------------------
1. EEPROM is physically accessible (I2C, standard pinout)
2. EEPROM contents are NOT cryptographically signed
3. VIP firmware does NOT verify EEPROM integrity (only CRC for corruption)
4. VIP transmits ICUSB status to SoC without authentication
5. HostOS passes IPC messages without content verification
6. Android gm_protokey trusts the flag value from VIP

18.4 ATTACK REQUIREMENTS
------------------------
Physical access to module for EEPROM modification:
- Remove module from vehicle
- Desolder or clip onto EEPROM (M24C64)
- Read/modify/write using programmer (XGecu T48, etc.)
- Reinstall module

No special tools needed:
- Standard I2C EEPROM programmer
- No cryptographic keys required
- No firmware exploits needed
- Works on any module (Y177, Y181, etc.)

18.5 SECURITY DESIGN FLAW
-------------------------
The fundamental flaw is trusting configuration data from an unprotected
external storage device. Proper design would:
- Store security flags in OTP (One-Time Programmable) memory
- Cryptographically sign EEPROM contents
- Verify signatures before trusting flags
- Use secure element for sensitive configuration

21.3 POTENTIAL SOFTWARE MITIGATIONS
-----------------------------------
GM could implement in VIP firmware update:

1. EEPROM Integrity Verification
   - Store hash of critical EEPROM regions in OTP/secure storage
   - Verify hash at boot
   - Refuse to boot if mismatch detected

2. Cryptographic Signing
   - Sign EEPROM contents with key stored in secure element
   - Verify signature before trusting flags

3. Runtime Tamper Detection
   - Periodically re-read and verify EEPROM
   - Alert if values change unexpectedly

4. Flag Lockdown
   - Move security-critical flags to OTP
   - Ignore EEPROM values for ICUSB after update

21.4 WHAT GM CANNOT FIX
-----------------------
1. Existing Modified Modules
   - Users who already modified EEPROM keep access
   - No way to force EEPROM restoration

2. Physical Hardware Vulnerability
   - EEPROM remains physically accessible
   - I2C interface cannot be disabled
   - Chip can always be desoldered and reprogrammed

3. Downgrade to Vulnerable Firmware
   - If user captures pre-fix firmware
   - Can re-flash older VIP firmware
   - Unless anti-rollback is enforced

21.5 RECOMMENDED GM ACTIONS
---------------------------
1. Immediate: Update VIP firmware with EEPROM hash verification
2. Medium-term: Move security flags to secure storage in new hardware
3. Long-term: Redesign security architecture for future vehicles

21.6 ATTACK PERSISTENCE
-----------------------
Even after GM OTA fix:
- Attacker can block OTA updates
- Attacker can re-modify EEPROM after update
- Attacker can downgrade VIP firmware (if anti-rollback not enforced)
- Physical access = persistent compromise

23.6 VIP-EEPROM DATA FLOW
-------------------------
Boot Sequence:

1. VIP Power-On Reset
   ↓
2. VIP Bootloader (BOLO) initializes
   ↓
3. RH850-IIC peripheral initialized
   ↓
4. EEPROM read via I2C:
   - Calibration values (cal_*)
   - Security flags (ICUSB at 0x0440)
   - VIN data (0x05C0)
   - Configuration data
   ↓
5. VIP Application starts
   ↓
6. PROTOKEY module reads ICUSB flag
   ↓
7. IPC connection to SoC established
   ↓
8. PROTOKEY status sent to SoC

EEPROM Access Pattern:
```
VIP reads EEPROM → Stores in RAM → Uses throughout operation
                     ↓
                 Sends to SoC via IPC
                     ↓
                 SoC acts on flags (gm_protokey daemon)
```

23.7 PROTOKEY-EEPROM-SOC RELATIONSHIP
-------------------------------------
Detailed ICUSB Flag Flow:

1. EEPROM Address 0x0440:
   Stock:    C3 00 C3 (Frame=0xC3, Value=0x00)
   Modified: 5A FF 5A (Frame=0x5A, Value=0xFF)

2. VIP Boot Process:
   [PROTOKEY] → Reads ICUSB flag from EEPROM
   [PROTOKEY] → If 0xFF: "ICUSB module enabled" + bypass mode
   [PROTOKEY] → If 0x00: Normal security mode

3. Seed Handling:
   BCM → GMLAN → VIP → Cache seeds
   VIP → Check ICUSB flag
   VIP → Transmit to SoC via IPC:
     - status=1: Valid seeds, require GM Secure Client
     - status=3: Empty/bypass, allow unsecured access

4. SoC (gm_protokey daemon):
   Receives PROTOKEY status from VIP via IPC
   If bypass mode: Allow any ADB connection
   If normal mode: Require GM RSA authentication

23.9 SECURITY IMPLICATIONS
--------------------------
1. RH850 has NO secure boot for EEPROM:
   - EEPROM contents trusted without verification
   - No cryptographic signing of calibration data
   - Only CRC for corruption detection

2. I2C is physically accessible:
   - Standard protocol, easy to intercept/modify
   - EEPROM can be desoldered and reprogrammed
   - No tamper detection

3. VIP blindly trusts EEPROM:
   - No integrity verification beyond CRC
   - Security flags directly control behavior
   - Modification persists across power cycles

4. AUTOSAR abstraction layer:
   - NvM_ functions read from EEPROM
   - Application code trusts NvM data
   - No security layer between storage and use

23.11 VIP FIRMWARE SECURITY FEATURES
-------------------------------------
Observed security mechanisms:

1. SBA Ticket Validation:
   - "SBA ticket is valid" / "SBA ticket is invalid"
   - Fails gracefully (soft failure mode)

2. Signed Firmware Updates:
   - "Application signed header is valid/invalid"
   - Checked during firmware update only

3. IPC Channel Validation:
   - "Got IPC msg on invalid channel"
   - "Invalid IPC message length"

4. Watchdog Timer:
   - "Bootloader timeout reset"
   - Prevents system hangs

Missing security:
- No EEPROM content signing
- No runtime tamper detection
- No secure boot chain for configuration

24.17 VIP ACCESS FROM RUNNING ANDROID SYSTEM
---------------------------------------------

The VIP (Renesas RH850 MCU) handles EEPROM, CAN bus, security functions.
Access paths from Android:

INTERFACE                 ACCESS LEVEL              NOTES
-------------------------------------------------------------------------------
/dev/ipc/ipc2-ipc20       SELinux protected         19 IPC channels to VIP
/dev/ipc/ctrlif_s         SELinux protected         Control interface socket
/dev/ttyACM1              crw-rw-rw- (world)        SELinux blocks shell write
IPCServer (PID 383)       runs as root              Main VIP IPC daemon
gm_protokey (PID 845)     runs as root              Security/auth handler
gm_vehicle_hal            runs as vehicle_network   VHAL → VIP bridge

To access VIP from Android requires:
  1. Compromise IPCServer, gm_protokey, or gm_vehicle_hal process
  2. Get SELinux permissive (Y177 downgrade or EEPROM 0x0B40 flag)
  3. Use a privileged app context with ipc_device access

The IPC protocol uses HDLC framing over Unix domain sockets. Each channel
(ipc2-ipc20) handles different message types. The ctrlif_s socket appears
to be the control/management interface.

24.18 STEERING WHEEL / MEDIA CONTROL ACCESS - CRITICAL FINDING
---------------------------------------------------------------

*** CONFIRMED: ADB input keyevent CONTROLS RADIO ***

Testing revealed that running:
  adb shell input keyevent KEYCODE_MEDIA_NEXT

Actually changes the radio station forward. This means:

CONTROL               DIRECT ACCESS    ADB/SHELL ACCESS    NOTES
-------------------------------------------------------------------------------
Media Next/Prev       NO (signature)   YES - WORKS!        Changes radio station
Volume Up/Down        YES (standard)   YES                 Normal Android behavior
Play/Pause            NO (signature)   YES                 Media control
Call Answer/Hangup    NO (signature)   YES (untested)      Phone integration
PTT Button            PARTIAL          YES                 See SWC_PTT_APP below

Permission Analysis:
  - gm.permission.RECEIVE_KEY_EVENT: signature|privileged (third-party blocked)
  - android.permission.INJECT_EVENTS: signature (third-party blocked)

However, ADB shell bypasses these restrictions because:
  1. Shell runs with elevated privileges
  2. input command uses InputManager directly
  3. No permission check for shell user

SECURITY IMPLICATIONS:

1. Any app with shell/root execution can control media:
   Runtime.getRuntime().exec("input keyevent KEYCODE_MEDIA_NEXT")

2. Accessibility services with PERFORM_ACTION capability could simulate keys

3. USB devices with HID capabilities could inject key events

4. Remote ADB access (if enabled) provides full media control

POTENTIAL EXPLOITS:

1. PTT Button Hijack (verified writable):
   adb shell settings put secure com.gm.vac.SWC_PTT_APP com.malicious.app

2. Media Control via Shell Execution:
   Any app exploiting a shell vulnerability can control radio/media

3. Accessibility Service Attack:
   Install app with AccessibilityService, user enables it, then:
   - performGlobalAction(GLOBAL_ACTION_MEDIA_NEXT) may work
   - dispatchGesture() for complex interactions

4. USB HID Injection:
   Physical USB device presenting as HID keyboard can send media keys

Apps with INJECT_EVENTS permission (all system apps):
  - com.android.car
  - com.gm.* (various)
  - android (system_server)

Enabled Accessibility Services:
  - com.android.car/.pm.CarSafetyAccessibilityService (stock)

24.19 VOICE ASSIST / PTT BUTTON - FULL EVENT CHAIN ANALYSIS

Physical button press captured via logcat reveals complete event flow:

CAN FRAME IDENTIFICATION:
  SWC KEY 1002 = Voice Assist / PTT button on steering wheel

  GMVHAL  (  771): SWC KEY 1002 EVENT 0   <- Button pressed (CAN → VHAL)
  GMVHAL  (  771): SWC KEY 1002 EVENT 1   <- Button released

KEY CODE MAPPING:
  Physical Button    → CAN Frame  → Android Key Code
  Voice Assist/PTT   → 1002       → KEYCODE_VOICE_ASSIST

  1. VIP (RH850) receives CAN frame from steering wheel module
  2. IPC channel passes to Android side
  3. gm_vehicle_hal receives via VHAL
  4. GMCarInputService (PID 1618) translates to Android KeyEvent
  5. PhoneProjectionService.vrKeyEvent() handles routing

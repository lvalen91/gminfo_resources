================================================================================
                         EXECUTIVE SUMMARY
================================================================================

GM vehicles with AAOS (Android Automotive OS) have multiple remote access
pathways that allow General Motors to:

  - Push firmware updates (potentially mandatory)
  - Access vehicle CAN bus remotely
  - Collect telemetry and security event data
  - Execute remote commands (start, lock, climate, etc.)

These capabilities exist through TWO independent systems:
  1. OnStar/Telematics Module (works WITHOUT the radio installed)
  2. A11 Infotainment Radio (AAOS)

================================================================================
               SECTION 5: USER CONSENT ANALYSIS
================================================================================

5.1 OPERATIONS REQUIRING USER CONSENT
-------------------------------------
  - USB firmware updates (CONSENT_PROCEED timeout: 11 seconds)
  - OTA updates (notification displayed)

5.2 OPERATIONS WITHOUT EXPLICIT CONSENT
---------------------------------------
  - Security Event Logging (always-on telemetry)
  - Device registration
  - Calibration value queries
  - Diagnostic data collection

5.3 POTENTIALLY CONSENT-BYPASSING
---------------------------------
  - "(mandatory)" flag in Red Bend UA suggests forced updates possible
  - gmConnectionService has REBOOT permission (can reboot without consent)
  - gmConnectionService has START_ACTIVITIES_FROM_BACKGROUND
  - WRITE_SECURE_SETTINGS allows modifying system settings

5.4 CALIBRATION FILE SECURITY RESET
------------------------------------
GM added security flag reset to calibration files (defense-in-depth):

File: 85783460 (SW ID 07)
Contains encrypted payload that resets:
  - EEPROM 0x0440 -> C300 (Security LOCKED)
  - EEPROM 0x0A80 -> C300 (Backup LOCKED)
  - EEPROM 0x0B40 -> 6900 (Debug mode OFF)

This means SPS calibration visits can reset security modifications.

               SECTION 6: TELEMATICS vs INFOTAINMENT
================================================================================

CRITICAL: OnStar/Telematics operates INDEPENDENTLY of A11 Radio!

+------------------------------------------------------------------+
| CAPABILITY              | TELEMATICS (CGM) | A11 RADIO (AAOS)    |
+------------------------------------------------------------------+
| Internet Connection     | LTE (AT&T)       | WiFi or via CGM     |
| CAN Bus Access          | DIRECT           | Via VIP (RH850)     |
| Remote Start            | YES              | NO (relay only)     |
| Remote Lock/Unlock      | YES              | NO (relay only)     |
| Works Without Radio     | YES              | N/A                 |
| User Can Disable        | NO*              | YES (sort of)       |
+------------------------------------------------------------------+
* Requires physical disconnection of telematics module

PROOF: Remote start, lock, and climate control work even with
A11 radio removed. This confirms telematics has independent:
  - Cellular modem
  - CAN bus connection
  - Command execution capability
[NOTE: this table is derived from STATIC firmware analysis and is consistent with
 OnStar's known design (telematics/CGM is independent of the AAOS infotainment radio).
 Runtime execution of remote commands was not directly observed in this analysis -- treat
 the "YES" cells as strong static-analysis inferences, not bench-verified behavior. See the
 disclaimer at the end of this document.]

================================================================================
            SECTION 9: GHS HYPERVISOR ARCHITECTURE
================================================================================

GM's A11 radio uses a GHS (Green Hills Software) Hypervisor that runs BELOW
Android, providing safety-critical functions independently of the OS.

9.1 HYPERVISOR DEVICE INTERFACES
--------------------------------
GHS provides multiple device interfaces to Android:

+------------------------------------------------------------------+
| DEVICE PATH          | SELinux CONTEXT       | PURPOSE            |
+------------------------------------------------------------------+
| /dev/ghs/ipc         | ghs_device            | Inter-Process Comm |
| /dev/ghs/cal         | ghs_device            | Calibration        |
| /dev/ghs/chime       | ghs_device            | Audio Chimes       |
| /dev/ghs/camera      | ghs_camera_device     | Camera Access      |
| /dev/ghs/tee-att     | ghs_device            | TEE Attestation    |
| /dev/ghs/audit       | ghs_device            | Security Audit     |
+------------------------------------------------------------------+

9.2 CAMERA BYPASS ARCHITECTURE
------------------------------
CRITICAL FINDING: Backup camera works DURING Android reboot!

When Android reboots (adb reboot, steering wheel hold), the display shows
the backup camera feed if the vehicle is shifted into reverse. This proves:

  1. GHS hypervisor continues running when Android restarts
  2. GHS receives CAN bus signals (gear position) independently
  3. GHS directly controls display overlay for camera feed
  4. Android EVS (External Vehicle Service) is a PASSTHROUGH, not the source

CAMERA DATA FLOW:
+------------------------------------------------------------------+
|                                                                  |
|  [Rear Camera] --> [Video Input] --> [GHS Hypervisor]            |
|                                           |                      |
|                                           | /dev/ghs/camera      |
|                                           v                      |
|  [CAN Bus: Gear Position] -------> [GHS Display Controller]      |
|                                           |                      |
|                                           | Direct Overlay       |
|                                           v                      |
|                                    [Display Panel]               |
|                                    /dev/ttyS3, /dev/i2c-7        |
|                                                                  |
+------------------------------------------------------------------+

9.3 EARLY EVS SYSTEM
--------------------
Binary: /vendor/bin/earlyEvs_harman (COMMENTED OUT in production)

SELinux capabilities of early_evs domain:
  - camera_device access (read/write)
  - video_device access (read/write)
  - gpu_device access (read/write)
  - display_panel_device access
  - can_socket access (CAN bus!)
  - serial_device access

The early_evs service was designed to provide camera during early boot
BEFORE Android fully starts. In production, this is handled by GHS directly.

9.4 WHO ACCESSES GHS CAMERA?
----------------------------
SELinux policy shows these domains access ghs_camera_device:

  - gmConnectionService  (GM's main connection service)
  - system_app          (System applications)

This allows Android apps to request camera overlays via GHS when needed.

9.5 DISPLAY PANEL CONTROL
-------------------------
Display panel devices (/dev/ttyS3, /dev/i2c-7) are accessed by:

  - gmConnectionService
  - gm_update_engine
  - gm_vehiclepanel
  - early_evs
  - remote_display
  - system_app

This multi-layered access confirms GHS can control display independently.

9.6 SECURITY IMPLICATIONS
-------------------------
  - GHS continues operating when Android crashes or reboots
  - Camera feed is a SAFETY feature - must work without Android
  - GHS has direct hardware access, bypassing Android security
  - GHS could theoretically display anything on screen without Android
  - CAN bus signals reach GHS before Android processes them

================================================================================
                      SECTION 10: KEY FINDINGS SUMMARY
================================================================================

1. GM has DUAL remote access paths (Telematics + A11 Radio)

2. Telematics module operates INDEPENDENTLY - removing the radio
   does NOT disable remote vehicle control

3. Red Bend Update Agent supports "(mandatory)" updates - potential
   for forced updates without user consent

4. gmConnectionService has REBOOT permission - can restart device
   without user interaction

5. Security Event Logging is ALWAYS ON - telemetry constantly
   reports to vehicle.api.gm.com:20445

6. A11 and Telematics share internal Ethernet network - data can
   flow between them

7. GIS768_IgnoreCurrentPartNumberCheck = TRUE bypasses the part-number check
   on updates ONLY -- it does NOT enable a firmware downgrade. The actual
   anti-rollback is the GHS hypervisor's rollback counter (misc/vda9), which
   blocks Y181->Y177 at boot (tested FAILED). See section 5.4 correction above.

8. Calibration files now include security reset payloads as
   defense-in-depth against EEPROM modifications

9. Complete remote access chain: Cloud -> Telematics -> Android
   Service -> Update Engine -> Red Bend -> Flash

10. No way to fully disable remote access without physical
    disconnection of telematics module

11. GHS Hypervisor operates BELOW Android, controlling camera and
    display directly - backup camera works even during Android reboot

12. GHS has direct CAN bus access and can overlay content on display
    independently of Android OS state
